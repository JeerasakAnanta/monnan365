## คำสั่งสำหรับ AI ที่จะสร้างสไลด์ (อ่านก่อนเริ่ม)

1. **เวลา**: นำเสนอ **10 นาที** + ตอบคำถาม **5 นาที** → ออกแบบสไลด์ประมาณ **10–12 แผ่น** (เฉลี่ย <1 นาที/แผ่น ยกเว้นสไลด์ Demo ที่ควรได้เวลามากกว่าเพื่อน — ประมาณ 2 นาที)
2. **ลำดับการนำเสนอ**: ทีมนี้ขึ้นพูด **ลำดับที่ 2 จาก 7 ทีม** ในวัน pitching — ต้องพร้อมเร็ว ไม่มีเวลา warm-up นาน
3. **ภาษา**: ไทยเป็นหลักตลอดทั้งเดค (deck) คงคำศัพท์ทางการที่ใช้ในเอกสารต้นฉบับไว้ เช่น "กรีนซีซัน (Green Season)", "วิสาหกิจชุมชน", "แหล่งท่องเที่ยวรอง", "ไร้ฤดู" — อย่าแปลหรือเปลี่ยนคำเหล่านี้
4. **โทน/ธีมภาพ**: ใช้ Nan Province color palette (ดูหมวด Fact Sheet ด้านล่าง) — โทนป่าเขียว/ทองอุ่น/ฟ้าน้ำ บนพื้นครีม ไม่ใช้สีดำสนิท
5. **หลัก anti-hallucination**: **ห้ามแต่งตัวเลข ชื่อ หรือข้อเท็จจริงที่ไม่มีอยู่ในเอกสารนี้** ถ้าสไลด์ไหนต้องการข้อมูลที่ไม่มีแหล่งอ้างอิง ให้คงไว้เป็น placeholder ที่มองเห็นได้ชัด (เช่น `[ใส่ตัวเลขที่นี่]`) แทนการเดา — หลักการนี้สะท้อนกลไกจริงของตัวโปรดักต์เอง (ดูสไลด์ 7)
6. **Demo link จริงที่ใช้ได้ตอนนี้**: https://monnan.jeerasakananta.dev/
7. **ที่มาโครงสร้าง 10 สไลด์ด้านล่าง**: มาจากคำแนะนำทางการของผู้จัดงานในเอกสารประกาศผลผ่านเข้ารอบ ("โครงสร้าง Slide Pitch ที่แนะนำ") — ให้ยึดหัวข้อและลำดับนี้เป็นหลัก ไม่ต้องคิดโครงสร้างใหม่

### สไลด์ 2 — Problem

ปัญหาการท่องเที่ยวจังหวัดน่านที่เลือกแก้ (คำต่อคำจาก README):

> การท่องเที่ยวจังหวัดน่านกระจุกตัวอยู่ในช่วงฤดูหนาวเพียงไม่กี่เดือน ทำให้
> - รายได้ของผู้ประกอบการและชุมชนขาดช่วงในฤดูอื่น
> - แหล่งท่องเที่ยวรองและวิสาหกิจชุมชนไม่เป็นที่รู้จัก แม้มีคุณค่าตลอด 12 เดือน
> - นักท่องเที่ยวไม่รู้ว่า "นอกฤดูหนาว น่านมีอะไร" — ทั้งที่ทุกเดือนมีเทศกาล วัฒนธรรม อาหาร และธรรมชาติเฉพาะฤดู

Speaker note: นี่คือปัญหาที่โจทย์ต้องการให้แก้โดยตรง ("กระจายมูลค่าการท่องเที่ยวน่านให้ครอบคลุมตลอด 12 เดือน") — ควรพูดเชื่อมกับ mission ของ hackathon ทันที

### สไลด์ 3 — User / Persona

> **[สังเคราะห์จากข้อมูลกระจัดกระจายในเอกสาร — ไม่มี persona doc ทางการ]**

- **ผู้ใช้หลัก**: นักท่องเที่ยวไทย (domestic) ที่วางแผนเดินทางไปน่าน แต่ไม่รู้ว่านอกฤดูหนาวมีอะไรให้เที่ยว
- **Pain point**: เห็นแต่คอนเทนต์ "น่านหน้าหนาว" ซ้ำๆ ไม่รู้จักแหล่งท่องเที่ยวรอง ไม่มั่นใจว่าเดือนอื่นคุ้มค่าเดินทางไหม
- **ผู้ได้ประโยชน์รอง (secondary beneficiaries)**: ผู้ประกอบการท้องถิ่น, วิสาหกิจชุมชน, โฮมสเตย์ — ที่ต้องการช่องทางให้นักท่องเที่ยวมองเห็นตลอดทั้งปี ไม่ใช่แค่ high season
- กลุ่มที่โจทย์ระบุว่าเป็นผู้เข้าร่วม hackathon (นักเรียน/นักพัฒนา/ผู้ประกอบการ/คนทำคอนเทนต์/บุคลากรท่องเที่ยว) เป็นคนละกลุ่มกับ end-user ของโปรดักต์ — อย่าสับสนสองกลุ่มนี้บนสไลด์

### สไลด์ 4 — Insight

ทำไมปัญหานี้สำคัญต่อ 12 เดือนของน่าน — เชื่อมกับเป้าหมายหลักของโจทย์ 4 ข้อ:

| เป้าหมายโจทย์ (น่าน ไร้ฤดู) | Insight |
|---|---|
| กระจายมูลค่าสู่ท้องถิ่น | รายได้กระจุกตัวฤดูหนาว = ชุมชนรองไม่ได้ประโยชน์เลยตลอด 8-9 เดือน |
| การลงมือทำด้วยเทคโนโลยี | ต้องใช้ AI/Design Thinking แก้ปัญหานี้แบบวัดผลได้ ไม่ใช่แค่แคมเปญประชาสัมพันธ์ |
| สร้างความน่าสนใจตลอด 12 เดือน | น่านมีของดีทุกเดือนอยู่แล้ว แค่ไม่มีใครสื่อสารให้นักท่องเที่ยวเห็น |
| พัฒนาด้วย Design Thinking | ต้นแบบต้องใช้งานได้จริงในชุมชนท้องถิ่น ไม่ใช่แค่ demo สวย |

ตัวอย่างคอนกรีตจาก dataset จริง:
- **วัดภูมินทร์** — `season_note: "ชมได้ทั้งปี ฝนตกก็เที่ยวได้"` → ตัวอย่างสถานที่ที่ underused นอกฤดูหนาว ทั้งที่เที่ยวได้ทุกฤดู
- **ดอยภูคา** — `season_note: "ดอกนางพญาเสือโคร่งบาน ม.ค."` → ตัวอย่าง high-season draw ที่ปกติดึงคนไปกระจุกแค่เดือนเดียว

### สไลด์ 5 — Solution

**MonNan365 คือ AI Trip Planner ที่วางแผนเที่ยวน่านตามเดือนที่ผู้ใช้จะเดินทางจริง — ไม่ใช่แนะนำแต่ที่ยอดนิยมหน้าหนาว**

ผู้ใช้เลือก: เดือนเดินทาง / จำนวนวัน (1–5) / สไตล์ (ธรรมชาติ, วัฒนธรรม, อาหาร, Wellness) / งบประมาณทั้งทริป (THB)

ระบบสร้างแผนเที่ยวรายวันด้วยกลไก 3 อย่างที่ทำให้ต่างจาก trip planner ทั่วไป:

1. **ขายจุดเด่นของฤดูนั้น** — เช่น Green Season มีนาข้าวเขียว หมอกหน้าฝน ที่พักถูกกว่า คนน้อยกว่า
2. **บังคับกระจายสู่ชุมชน** — ทุกแผนมีแหล่งท่องเที่ยวรอง/วิสาหกิจชุมชนอย่างน้อย **~35%** พร้อมช่องทางติดต่อจริง
3. **AI อธิบายเหตุผล** — ทุกจุดหมายมีคำอธิบายว่าทำไมเหมาะกับเดือนและสไตล์ของผู้ใช้

ตาราง "ความสอดคล้องกับเป้าหมายโจทย์" (จาก README, ใช้ได้ตรงตัวบนสไลด์นี้หรือสไลด์ Insight):

| เป้าหมายโจทย์ | สิ่งที่ MonNan365 ทำ |
|---|---|
| กระจายนักท่องเที่ยวออกจากฤดูหนาว | Planner เชิงรุกที่ชูจุดเด่นของ Low/Green Season |
| กระจายรายได้สู่ชุมชนรอง | Quota แหล่งท่องเที่ยวรอง + ข้อมูลติดต่อวิสาหกิจชุมชน |
| เพิ่มเวลาพำนักและกิจกรรม | แผนรายวันหลายวัน ผสมกิจกรรมหลากประเภท |
| ยกระดับดิจิทัลท้องถิ่น | โครงสร้างข้อมูลเปิด (JSON) ที่ชุมชนต่อยอดเพิ่มรายการเองได้ |

### สไลด์ 6 — Demo Flow

แนะนำ 1-2 flow หลักที่กรรมการควรเห็น (ควรได้เวลามากกว่าสไลด์อื่น ~2 นาที):

**Flow 1 — สร้างแผนเที่ยวเดือน Green Season (เช่น สิงหาคม)**
1. เลือกเดือน: สิงหาคม / จำนวนวัน: 2-3 วัน / สไตล์: ธรรมชาติ+วัฒนธรรม / งบ: กลาง
2. กดวางแผน → เห็นสรุป "ทำไมเดือนนี้ถึงน่าไปน่าน" + `low_season_perks` (เหตุผลว่าทำไม low season คุ้มกว่า)
3. เห็น timeline card รายวัน — ชี้ badge "Low Season" / "ชุมชน-แหล่งท่องเที่ยวรอง" บนแต่ละจุดหมาย พร้อมเหตุผลที่ AI อธิบาย
4. เปิดแผนที่ Leaflet ดูตำแหน่งจริง

**Flow 2 (ถ้าเวลาเหลือ) — Share/Print**
- กด Share → ได้ลิงก์แชร์แผนทริป (บันทึกลง Supabase) หรือ Print → PDF export พร้อม branding

Speaker note: เน้นให้กรรมการเห็น badge ชุมชน/แหล่งท่องเที่ยวรอง ชัดๆ เพราะตรงกับเกณฑ์ "Impact เชิงท่องเที่ยว" และ "ความเข้าใจชุมชน" โดยตรง (ดู Fact Sheet)

### สไลด์ 7 — AI / Data

**สถาปัตยกรรม: RAG-lite** — LLM ไม่เคยแต่งสถานที่เอง เลือกและอธิบายได้เฉพาะจาก dataset ที่ตรวจสอบแล้วเท่านั้น

```
User Input (เดือน, จำนวนวัน, สไตล์, งบ)
   ↓
/api/plan (Next.js API Route)
   ├─ 1. Supabase query: attractions ที่ months_best ครอบคลุมเดือนที่เลือก
   ├─ 2. selectAttractions(): กรองตามสไตล์ + บังคับ quota ชุมชน ≥35% + สุ่ม (Fisher-Yates)
   ├─ 3. เรียก OpenRouter API พร้อม zodResponseFormat (structured output บังคับ schema)
   │      - System prompt (ไทย, กติกา anti-hallucination): ใช้เฉพาะสถานที่ใน dataset, อ้างอิงด้วย id ตรงตัว,
   │        ทุกวันต้องมีแหล่งท่องเที่ยวรอง/ชุมชนอย่างน้อย 1 แห่ง, อธิบายเหตุผลที่เหมาะกับเดือน/สไตล์
   └─ 4. Enrich: map ID ที่ AI เลือกกลับเป็นข้อมูลสถานที่เต็ม (ชื่อ, พิกัด, contact, รูป)
   ↓
Timeline cards + แผนที่ Leaflet
```

- **โมเดล**: `google/gemini-3.5-flash` ผ่าน OpenRouter *(ค่าที่ยืนยันจากโค้ดจริงในระบบ ไม่ใช่ค่าที่เขียนไว้ใน README เดิม)*
- **Validation**: Zod + Structured Outputs — บังคับ schema ตั้งแต่ระดับ API ป้องกัน JSON เพี้ยน
- **Retry**: สูงสุด 3 ครั้งต่อโมเดล, exponential backoff
- **Dataset**: 40 รายการคัดสรร ครอบคลุม 10 อำเภอ ครบทั้ง 12 เดือน — **31 รายการเป็นแหล่งท่องเที่ยวรอง, 21 รายการเป็นวิสาหกิจชุมชน**
- **Selection algorithm** (`selectAttractions.ts`): แบ่ง pool หลัก/รอง → บังคับสัดส่วนขั้นต่ำ 35% จาก pool รอง/ชุมชน → สุ่มแบบ Fisher-Yates → ผสานผลลัพธ์

Speaker note: quota 35% ไม่ใช่แค่ prompt engineering ฝั่งเดียว แต่บังคับใน business logic ก่อนส่งเข้า LLM ด้วย — เป็นจุดที่ตอบโจทย์ "การใช้ AI" และ "ความเข้าใจชุมชน" พร้อมกัน

### สไลด์ 8 — Impact & Metrics

> ผลิตภัณฑ์เพิ่งเปิดใช้งานสำหรับ hackathon นี้ **ยังไม่มีตัวเลขการใช้งานจริง (usage data) ที่นำมาอ้างอิงได้** ณ ตอนนี้ — สไลด์นี้จึงเน้น **design-intent metrics** คือกลไกที่ออกแบบไว้เพื่อ "การันตี" ผลลัพธ์ ไม่ใช่ผลลัพธ์ที่วัดจากผู้ใช้จริงแล้ว

กลไกที่วัดผลได้จริงในตัวโปรดักต์ (ไม่ใช่แค่คำสัญญา):
- **35% secondary/community quota ถูกบังคับใน logic** ทุกแผนที่สร้างขึ้น — วัดได้ 100% ของ output ว่ามีสัดส่วนกระจายจริงหรือไม่
- แต่ละ badge "Low Season" / "ชุมชน-แหล่งท่องเที่ยวรอง" ที่แสดงบน UI คือหลักฐานเชิง traceable ว่าแผนไหนกระจายไปที่ไหนบ้าง

เทียบกับตัวชี้วัดที่ผู้จัดงานแนะนำ (ตัวชี้วัดความสำเร็จของผลงาน):
| ตัวชี้วัดจากโจทย์ | สถานะปัจจุบันของ MonNan365 |
|---|---|
| ปริมาณทริปช่วง Low Season | กลไก quota รองรับแล้ว แต่ยังไม่มีข้อมูลการใช้งานจริงมานับ |
| กระจายเงินสู่ชุมชนรอง | มี contact จริงของวิสาหกิจชุมชนแนบทุกแผน — ยังไม่มีข้อมูล conversion จริง |
| เพิ่มเวลาพำนัก & กิจกรรม | รองรับแผนได้สูงสุด 5 วัน หลายกิจกรรมต่อวัน |
| ยกระดับขีดความสามารถดิจิทัล | โครงสร้างข้อมูลเปิด (JSON) ให้ชุมชนต่อยอดเองได้ในอนาคต |

### ข้อกำหนดที่ควรพูดถึงได้ถ้าถูกถามใน Q&A

- **Responsible AI**: AI สร้างแผนจาก dataset ที่ตรวจสอบแล้วเท่านั้น ระบบสั่งห้ามแนะนำสถานที่นอกฐานข้อมูลเพื่อป้องกัน hallucination
- **ไม่มีระบบ login สำหรับผู้ใช้ทั่วไป** — กรรมการเข้าใช้งาน demo ได้ทันทีโดยไม่ต้องมี test account
- **ไม่มี API key/secret หลุดในโค้ด public** — ใช้ environment variables + `.gitignore`
- **Accessibility**: มีปุ่มปรับขนาดตัวอักษร (FontSizeToggle), รองรับมือถือ — ตรงกับเกณฑ์ bonus +5
- **ขอบเขตที่ตัดออกโดยตั้งใจ (MVP scope)**: ไม่มีระบบจอง/ชำระเงินจริง, ไม่ใช้ full vector DB/RAG (ใช้ "RAG-lite" filter+context แทน), รองรับภาษาไทยเท่านั้นในเวอร์ชันนี้ — เหตุผล: ลด infra ที่ต้อง deploy/debug ในเวลาจำกัดของ hackathon
